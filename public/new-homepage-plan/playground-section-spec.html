Playground Section — Technical Spec

priyamwada.me · Gallery of Experiments · Last updated June 2026

Concept

The Playground section is a 3D world the user can freely pan through — up, down, left, right, in every direction. It lives inside the "Gallery of Experiments" room on the homepage floor plan. When the user clicks into that room, they transition into this 3D space.

Each experiment/project in the playground floats in this 3D space as an object. The user navigates around, discovers projects, clicks on one and gets a lightweight card overlay with just enough information to understand what it is. No heavy case study structure, no long reads. Just a quick glance and a link out if there's something live to see.

Important scope boundary Full case study pages are separate pages entirely. The 3D world is only for playground/experiment projects, not the main portfolio case studies (Tars, Rocket Mortgage, Salesforce Galileo). Keep this page scoped tightly.

Tech Stack

Layer Tool Why
3D Rendering Three.js Standard WebGL abstraction. Well documented, large community, handles everything needed here.
Camera controls Custom mouse-follow camera OR OrbitControls from Three.js OrbitControls is quick to set up. Custom gives the cinematic feel where camera follows cursor direction instead of click-drag. Start with OrbitControls, upgrade if it feels clunky.
Click detection THREE.Raycaster Built into Three.js. Shoots a ray from camera through mouse position, detects which object was hit.
Post-processing EffectComposer + UnrealBloomPass Adds the glow/bloom on light points. Makes the scene feel cinematic rather than flat.
Overlay card HTML/CSS div positioned over the canvas Easier than 3D billboarded UI for readable text. Canvas blurs behind it via CSS backdrop-filter.
Smooth transitions gsap or manual lerping For camera movement toward a clicked object. Lerp is fine for simple cases; GSAP gives more control.

Scene Setup

Camera

Use a PerspectiveCamera. FOV around 60–75. The camera should feel like it's floating in space, not sitting on a ground plane.

Background

Deep black or very dark navy. #050508 or similar. The floor plan homepage is warm beige so this transition into darkness should feel like stepping into a different world.

Lighting

Minimal ambient light. Let project objects be self-lit (emissive materials) or use point lights anchored near each object. The background particle system handles the "light streaks" visual.

Background particle system

A BufferGeometry particle field with thin lines or points to create the grid/light-streak aesthetic seen on the Shopify reference. This is purely decorative, cheap to render and high visual impact. Make it slowly drift or rotate to give the world a sense of being alive.

Project Objects in 3D Space

Each playground project is represented as a floating object in the scene. The object can be a plane with a screenshot/preview texture on it, a custom shape, or an abstract form. Position them at varied depths (z-axis) and spread on x/y so the user has a reason to navigate around.

Object materials

  • Use MeshBasicMaterial or MeshStandardMaterial depending on whether lighting is needed.
  • For the wireframe aesthetic (like the Shopify reference), use wireframe: true on the material or a custom edge shader.
  • Objects should gently float/bob using a sine wave on the y-position in the animation loop.

Hover state

  • On hover (raycaster intersects object), scale the object up slightly: scale.set(1.08, 1.08, 1.08) with lerping so it eases in.
  • Change cursor to pointer.
  • Optionally add a subtle glow by increasing emissive intensity.

Click Interaction Flow

1
User clicks a project object

Raycaster detects the intersection. Grab the project data from the object's userData property.

2
Camera lerps toward the object

Move the camera closer and center the object in view. Use GSAP or manual lerp over ~600ms. This gives the user a moment of orientation before the card appears.

3
World softly blurs

Apply a CSS filter: blur(4px) or backdrop-filter to a transparent overlay div that sits on top of the canvas. This signals mode change to the user.

4
Card fades in

The project card HTML element fades up from opacity 0. It's centered on screen or anchored near the object's projected 2D position.

5
User dismisses

Click outside the card, click a close button, or press Escape. Card fades out, blur lifts, camera lerps back to its previous position. Resume panning.

Keep this light. These are playground experiments, not case studies. The card should be readable in under 10 seconds.

Entry Transition (from Homepage)

The user is on the homepage floor plan and clicks the "Gallery of Experiments" room. The transition into the 3D world should feel like stepping through a doorway, not a hard page load.

Options in order of complexity:

  • Simple: Fade to black, then fade into the 3D scene. Low effort, still works.
  • Better: The room on the floor plan zooms in (CSS transform scale) until it fills the viewport, then the 3D scene fades in over it.
  • Best: A brief fly-through animation plays on scene load, as if the camera is entering the space. The world starts dark and objects emerge as the camera moves forward.
Build order recommendation Get the 3D scene working with static object placement and the card overlay first. Add the entry transition last. It's the most delicate part and easiest to break if the scene isn't stable yet.

Performance Notes

  • Keep polygon count low on project objects. Planes and simple geometries only unless an object has a specific visual reason to be complex.
  • Limit post-processing passes. Bloom is expensive. One pass is fine, two starts hurting mobile.
  • If using video previews inside cards, lazy load them. Don't autoplay until the card opens.
  • Test on a mid-range laptop, not just a dev machine. The ocean shader on the homepage already uses GPU resources. This page adds more.
  • Add a fallback message if WebGL is not available: renderer.domElement creation will fail gracefully if you wrap it in a try/catch.
